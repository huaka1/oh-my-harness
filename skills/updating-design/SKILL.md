# 设计回流

创建或更新 `docs/harness/design/` 下的设计文档，使其反映实际构建了什么，而非初始计划猜测会发生什么。

设计文档不是 Agent 规则，也不是需求历史。它是人和模型理解项目事实背景的唯一事实源：先读方案文档；只有当方案多到需要导航时，才读索引。

Design 的默认产物是"方案说明书 + 当前事实架构"，不是摘要。每篇 design 应解释背景、领域边界、概念、流程、实现要点、决策原因、边界限制、代码方法锚点和验证方式，让新人能接手，让 Agent 能直接定位应该阅读和修改的代码。

Markdown 是主产物和唯一事实源。

此 skill 由用户触发。不要因为代码变更了或用户说他们会手动提交就自动运行。

<HARD-GATE>
1. 绝对禁止无 git diff 时编造变更。`git diff --stat` 为空时告知用户"没有未提交的变更。设计文档是最新的。"并停止。
2. 绝对禁止覆盖 standard/ 文件。`docs/harness/standard/` 下的文件只追加，不修改或删除已有内容。
3. 绝对禁止 git 写入操作。不执行 git add、git commit、git push。可读取 git diff、git log、git status。
4. 绝对禁止自动调用其他 skill。更新完成后报告并停止。
5. 只写 `docs/harness/design/` 和按知识去向追加 `docs/harness/standard/`；不得修改 workflow.md、feature/、plan/ 等设计回流工作面之外的文件。
6. design 文件名不得含日期、需求名、函数名；`docs/harness/design/` 下不得出现 `feature/`、`plan/` 子目录。
</HARD-GATE>

## 设计的单位

**设计的单位 = 一个长期机制/方案，对应一篇方案文档。** 其余一切（顶层 README、领域 README）都是懒创建的导航，有门槛才存在。

先有内容，再有目录；目录是为了导航已有内容，不是为了占位。

## 输入

- 活跃计划在 `docs/harness/plan/*.md`
- 当前 git diff 和最近提交
- 方案设计在 `docs/harness/design/<domain>/<mechanism>.md`
- 顶层入口（懒创建）在 `docs/harness/design/README.md`
- 领域索引（懒创建）在 `docs/harness/design/<domain>/README.md`
- 不使用日期或需求名作为 design 文件名
- 相关规范在 `docs/harness/standard/`
- feature 文档归 `docs/harness/feature/`，plan 归 `docs/harness/plan/`，design 只收长期机制方案

如果有多个活跃计划，问用哪个。如果没有活跃计划，说明没有可对工作面，询问用户是否要直接从 `git diff` 生成设计。

## Bug 定义

Bug 是任何发现的预期与实际行为之间的不匹配，且改变或阻塞了实现：

- 失败的测试、运行时错误、回归、损坏的 UI 流程、不正确的 API 行为
- 请求、计划或实现方法中的错误假设
- 在工作期间发现的依赖、平台、权限、数据或环境约束
- 应成为未来标准的重复 AI/操作者错误

临时错误在计划活跃时属于计划。只有持久知识才能存活到设计或标准中。

## 质量标准

每篇方案 design 必须同时满足两类读者：

- **人类读者**：5 分钟内能知道它解决什么问题、属于哪个领域、边界在哪里、主流程怎么走、失败时如何表现。
- **Agent 读者**：能从文档直接找到入口文件、关键类/函数、调用链、配置项、测试文件和修改时必须保护的边界。

质量门槛分两档，按方案粒度机械判定：

### 基础档（所有方案必须满足）

不满足以下任一项时，不要报告完成，继续补文档：

- 有目标读者或阅读目的说明。
- 有领域和领域边界，明确"负责什么"和"不负责什么"。
- 有流程说明（文字或简图皆可），说明请求、数据、状态如何流动。
- 有代码方法锚点，列出入口、关键函数/类、配置、测试和验证命令。
- 有关键决策表，包含原因和被放弃的替代方案。
- 有边界和限制，说明不支持场景、降级方式和未来可能改进。

### 增强档（方案跨 ≥2 个组件或 ≥2 个领域时追加强制）

- 有架构/领域分层图，说明模块依赖方向。
- 有关键时序图，说明跨组件的请求、数据、状态如何按时间流动。

判定方法：方案涉及的代码入口数或所属领域数。≥2 时进入增强档。

## 流程

1. **查找上下文**
   - 读取相关 `docs/harness/standard/*.md`
   - 找到活跃的 `docs/harness/plan/*.md`
   - 找到现有 `docs/harness/design/README.md`（若已懒创建）和相关领域/方案设计

2. **检查实现现实**
   - 运行 `git diff`
   - 运行 `git diff --staged`
   - 运行 `git log --oneline -10`
   - 在编辑设计之前总结实际变更了什么

3. **分类计划发现**
   - 与实现现实匹配的已完成 task：如果它们定义了功能，在设计中总结
   - 改变了方向的已完成 task：在设计中记录最终决策和原因
   - 未完成的 task：不纳入设计，除非它们是已发布行为的已知限制
   - 改变了决策的 bug：用最终决策和原因更新设计
   - 揭示了可复用实践的 bug：添加简短教训到 `docs/harness/standard/lessons.md`
   - 无未来价值的临时 bug：计划保留，不写入设计

4. **创建或更新设计**
   - 第一篇方案就放进对应领域目录：`design/<domain>/<mechanism>.md`，路径稳定不搬家
   - 领域归属在写第 1 篇时就是已知的，不先扁平再归类
   - 按长期机制/方案命名，例如 `browser/multi-tenant-isolation.md`、`browser/sso-auth-injection.md`、`auth/tgt-background-api.md`
   - 不按日期、需求名、函数名创建 design 文件
   - 重写过时部分而非追加矛盾历史
   - 用事实现在时写
   - 优先写背景、目标读者、领域边界、基础概念、流程、实现要点、设计决策、边界限制、代码方法锚点和验证方式
   - 可以引用关键代码片段和文件路径作为证据，但不要粘贴大段实现
   - 代码位置不要只写文件名；必须尽量写到类、函数、hook、配置键、测试文件和验证命令

5. **懒创建导航索引**
   - 顶层 `docs/harness/design/README.md`：全库 ≥3 篇方案，或 ≥2 个领域时才创建。它兼任全景入口，写系统全景、核心链路、主要领域和阅读路径。
   - 领域 `docs/harness/design/<domain>/README.md`：该领域 ≥3 篇方案时才创建，写本领域边界、方案清单和阅读顺序。
   - 门槛用篇数机械判定，不靠主观判断"有没有交叉引用"。
   - 跨门槛的那次更新顺手建索引，不回头补历史。第 3 篇落盘时同一次运行建索引；之后只更新不新建。
   - 未达门槛时不建索引，即使项目从未有过索引。

6. **消费 resolved bug 的知识去向**
   - 扫描计划文件的 `## Bugs and Findings` 区
   - 处理所有 `[x]`（已解决）bug 的 `知识去向` 字段：
     - `design` → 在对应 design 文件中记录决策和原因
     - `standard` → 追加到 `docs/harness/standard/` 下对应文件（只追加不覆盖），格式：
       ```markdown
       ## YYYY-MM-DD

       - [教训] 从 bug 中提炼的简短规则。上下文：为什么重要。
       ```
     - `drop` → 不写入任何地方
   - 只处理 `[x]`（已解决）bug。`[~]`（活跃）bug 不处理。

7. **质量自检**
   - 检查每篇新增或更新的方案 design 是否满足基础档标准
   - 跨 ≥2 组件/领域的方案检查是否满足增强档标准
   - 检查代码方法锚点是否能把读者带到真实实现位置
   - 检查图和表是否解释了领域边界、时序和关键决策，而不是装饰

8. **报告**
   - 说明创建或更新了哪个设计文件
   - 说明添加了哪些教训
   - 说明有多少 bug 的知识被消费（分别写入了哪些位置）
   - 说明是否新建或更新了导航索引（顶层 README / 领域 README）；未达门槛时说明"不适用"即可
   - 不要 commit，除非用户明确要求

## 设计模板

### 顶层入口索引模板（懒创建，全库 ≥3 篇或 ≥2 领域时才建）

```markdown
# Design Map

这是项目设计文档入口，兼任系统全景。需要理解项目背景时，先读本文件，再按领域进入具体方案文档。

## 系统全景

系统目标、核心链路、主要领域、关键约束的概述。

## 领域

- `browser/`：浏览器运行、CDP、用户隔离、SSO 注入、登录态刷新
- `auth/`：用户身份、登录态、TGT 获取、签名

## 阅读路径

- 要理解浏览器免登：读本文件 → `browser/sso-auth-injection.md` → `auth/tgt-background-api.md`
- 要理解多租户隔离：读本文件 → `browser/multi-tenant-isolation.md`
```

### 领域索引模板（懒创建，该领域 ≥3 篇时才建）

```markdown
# [Domain] Design

本领域负责什么，边界在哪里，不负责什么。

## 领域边界

- 负责：
- 不负责：
- 上游依赖：
- 下游影响：

## 方案文档

- `mechanism.md`：一句话说明解决什么问题

## 代码入口
```

### 方案设计模板

```markdown
# [机制/方案] 方案

> **目标读者**：开发 review、新人接手、Agent 理解上下文。本文档描述是什么、为什么、怎么分层、数据怎么流、代码方法在哪里。

## 背景

为什么需要这个机制？解决什么问题？

## 领域和边界

本方案属于哪个领域？负责什么？不负责什么？依赖哪些上游？影响哪些下游？

## 基础概念

读者理解方案前必须知道哪些组件、术语、外部系统？无新概念时可省略本节。

## 流程

请求、数据、状态如何流动？文字或简图皆可。跨组件的方案在此节内或追加节补架构/领域分层图和关键时序图。

## 实现要点

改了哪些模块，关键逻辑是什么？引用关键代码位置或少量代码片段。

## 代码方法锚点

| 类型 | 位置 | 作用 | 修改注意 |
|------|------|------|----------|
| 入口 | `path/to/file.py::entry()` | 谁调用本方案 | 入口参数和调用时机 |
| 核心逻辑 | `path/to/file.py::method()` | 主要决策在哪里发生 | 保持边界和副作用 |
| 配置 | `config/*.yaml#key` | 控制开关或参数 | 默认值和环境差异 |
| 测试 | `tests/...` | 证明行为 | 变更后必须运行 |

## 设计决策

| 决策 | 原因 | 替代方案 |
|------|------|----------|
| ... | ... | ... |

## 边界和限制

| 限制 | 影响 | 降级/未来改进 |
|------|------|----------------|
| ... | ... | ... |

## 验证方式

如何证明方案有效？列出命令、手工主链路、日志、trace、截图或其他证据入口。
```

## 粒度准则

- 一个长期机制或可独立解释的方案 = 一篇 design
- 例如：`multi-tenant-isolation.md`、`sso-auth-injection.md`、`tgt-background-api.md`
- 不要每个需求一篇 design；需求只是触发更新
- 不要把 design 写成 50 行摘要；缺少背景、概念、决策原因和验证方式时不算完整设计
- 不要按函数拆文档；函数细节留在代码里，design 写机制和边界，但必须给出关键方法锚点
- HTML 不是 design 默认产物；如用户明确要求展示层，作为一次性任务处理，不写进默认流程
- `docs/harness/design/` 下不得出现 `feature/`、`plan/` 子目录；feature 归 `docs/harness/feature/`，plan 归 `docs/harness/plan/`

### 领域摘要反例

以下不是完整 design，只是摘要，不要作为默认产物：

```markdown
# Browser 设计

Browser 负责 Chrome、CDP、SSO、隔离。

## 当前行为

几句描述当前行为。
```

它缺少背景、基础概念、流程、改造详解、验证方法和代码证据。

### 代码位置反例

以下不够用：

```markdown
## 代码位置

- `assistant/emotion/`
- `observability/`
```

它无法帮助 Agent 定位入口和修改面。应写到文件、类/函数、配置键、测试和验证命令，填入"代码方法锚点"表格，而不是散列目录路径。

## 常见错误

- 把初始计划当成真相。计划是证据，不是权威。
- 永远记录每个 bug。只有持久的设计决策和可复用的教训才能存活。
- 太早更新设计。如果实现还在快速变化，在计划中保留笔记直到用户要求对账。
- 自动 commit。此 skill 准备事实设计更新；用户可能手动 commit。
- 把 `CLAUDE.md` 当 design。`CLAUDE.md` 只放 Agent 工作规则；项目事实背景放 `docs/harness/design/`。
- 把每个需求都变成一篇 design。需求是触发更新，design 要按长期领域沉淀。
- 一上来就建顶层 README 和领域 README。索引是懒创建的，未达门槛不建。
- 强制单组件方案凑分层图和时序图。增强档只对跨组件/跨领域方案强制。
- 把 HTML 当事实源或默认产物。HTML 不是 design 的默认产物。
- 把 feature/plan 文档塞进 design 目录。feature 归 `docs/harness/feature/`，plan 归 `docs/harness/plan/`。
- 只写给人看，不写代码方法锚点。Design 也必须帮助 Agent 找到入口、调用链、配置和测试。
- 只写给 Agent 看，堆方法名而不解释领域和流程。Design 也必须让人能建立系统模型。
